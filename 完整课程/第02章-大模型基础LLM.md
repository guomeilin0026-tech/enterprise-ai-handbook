# 第 2 章：大模型基础（LLM）

## 2.1 什么是大模型

大模型可以理解成一个强大的文字理解和生成系统。

它能做：

- 回答问题
- 总结文章
- 改写内容
- 写代码
- 分析文本
- 提取信息

但它不能天然做到：

- 知道你的公司内部资料
- 直接查你的数据库
- 保证永远正确
- 自动执行真实业务操作

所以企业 AI 应用需要 RAG、工具调用、权限、安全和工程化。

## 2.2 消息角色

聊天模型通常有三类消息：

```text
system：规则和身份
user：用户输入
assistant：模型回答
```

例子：

```json
[
  {"role": "system", "content": "你是企业知识库助手"},
  {"role": "user", "content": "年假怎么算？"}
]
```

## 2.3 Token

Token 是模型处理文本的计量单位。

它影响：

- 费用
- 速度
- 上下文容量
- 能放多少聊天记录和资料

企业项目必须统计 Token，因为模型调用是有成本的。

## 2.4 上下文窗口

模型每次回答时，只能看到你传给它的上下文。

上下文可能包括：

- System Prompt
- 历史聊天记录
- 当前问题
- RAG 检索资料
- 工具调用结果

模型不是永久记忆，它是根据这次输入来回答。

## 2.5 Temperature

Temperature 控制回答随机性。

```text
低 temperature：更稳定，适合企业问答、结构化输出
高 temperature：更发散，适合创意写作
```

企业应用一般优先稳定。

## 2.6 流式输出

普通输出：

```text
等模型全部生成完，再一次性返回
```

流式输出：

```text
模型生成一点，页面显示一点
```

聊天产品通常使用流式输出，因为体验更好。

## 2.7 幻觉

幻觉就是模型编造不存在的信息。

常见原因：

- 问题缺少资料
- Prompt 不严格
- 模型被迫回答
- 上下文里有冲突信息

减少幻觉的方法：

- 使用 RAG
- 要求引用来源
- 资料不足时允许回答不知道
- 对关键结果做校验
