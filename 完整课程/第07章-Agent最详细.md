# 第 7 章：Agent（最详细）

## 本章目标

学完本章，你要能回答：

- Agent 是什么
- Agent 和普通聊天有什么区别
- Agent 和 RAG 有什么区别
- Agent 如何调用工具
- 企业 Agent 为什么必须做权限、日志和人工确认

## 7.1 Agent 是什么

Agent 是能根据用户目标，自主选择步骤和工具来完成任务的 AI 系统。

普通聊天：

```text
用户问一句
AI 回答一句
```

Agent：

```text
用户提出目标
AI 判断下一步要做什么
必要时调用工具
读取工具结果
继续推理
最终完成任务
```

一句话：

```text
普通 AI 会说，Agent 会调用工具做事。
```

## 7.2 Agent 的组成

一个企业 Agent 通常包含：

- 大模型
- System Prompt
- 工具列表
- 工具调用协议
- 权限系统
- 工具调用日志
- 任务状态
- 失败兜底
- 人工确认机制

## 7.3 Agent 工作流程

```text
用户输入任务
-> 后端把可用工具告诉模型
-> 模型判断是否需要工具
-> 模型生成工具名和参数
-> 后端校验权限和参数
-> 后端执行工具
-> 工具结果返回给模型
-> 模型判断是否继续调用工具
-> 模型输出最终回答
```

例子：

```text
用户：帮我查订单 10086 的物流状态
AI：需要调用 get_order_status
后端：校验用户是否有权限查这个订单
工具：返回已发货，预计明天送达
AI：订单 10086 已发货，预计明天送达
```

## 7.4 Agent 和 RAG 的区别

RAG 重点是查资料。

Agent 重点是做任务。

对比：

```text
RAG：根据员工手册回答年假怎么算
Agent：查询你的年假余额，并帮你发起请假申请
```

企业里经常组合：

```text
RAG + Agent
```

例如：

```text
先用 RAG 查报销制度
再用 Agent 查询发票信息
最后判断能不能报销
```

## 7.5 工具是什么

工具就是后端提供给 AI 使用的能力。

例如：

```text
get_order_status
search_knowledge_base
create_ticket
generate_report
send_email_draft
```

工具本质上是后端函数或接口。

AI 不直接操作数据库，AI 只是决定：

```text
我要调用哪个工具，参数是什么。
```

真正执行的是后端。

## 7.6 工具风险分级

### 低风险工具

只查询，不修改数据。

例如：

- 查询订单
- 查询知识库
- 查询用户信息

### 中风险工具

会创建草稿或生成内容，但不直接生效。

例如：

- 生成报表
- 创建工单草稿
- 生成邮件草稿

### 高风险工具

会影响真实业务。

例如：

- 删除数据
- 正式发送邮件
- 发起付款
- 提交审批

高风险工具必须人工确认。

## 7.7 企业 Agent 必须具备的能力

### 权限控制

用户只能调用自己有权限的工具。

### 参数校验

不能信任模型生成的参数。

### 日志审计

必须记录：

- 谁调用
- 调用哪个工具
- 参数是什么
- 返回什么
- 是否成功

### 失败兜底

工具失败时要告诉用户原因，而不是胡说。

### 人工确认

高风险操作必须暂停，等待用户确认。

## 7.8 Agent 常见架构

```text
Vue3 前端
-> FastAPI Agent 接口
-> 大模型判断工具
-> 工具执行器
-> 企业业务系统
-> 工具日志
-> 大模型总结
-> 前端展示
```

## 7.9 前端如何展示 Agent

不要只展示最终答案，最好展示过程：

```text
正在分析用户任务...
准备调用工具：查询订单
工具参数：order_id = 10086
工具返回：已发货
最终回答：订单已发货，预计明天送达
```

这样用户更信任系统。

## 7.10 常见坑

### 坑 1：工具太多

初期不要给 Agent 太多工具，工具越多越容易选错。

### 坑 2：没有权限控制

不能让 AI 代替后端权限系统。

### 坑 3：没有人工确认

高风险操作自动执行非常危险。

### 坑 4：把 Agent 当万能系统

Agent 适合复杂任务，但简单固定流程可能 Workflow 更稳定。

你只需要记住：

```text
Agent = 大模型 + 工具 + 决策流程 + 权限控制。

```
