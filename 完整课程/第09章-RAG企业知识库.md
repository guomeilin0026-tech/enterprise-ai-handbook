# 第 9 章：RAG（企业知识库）

## 本章目标

学完本章，你要能回答：

- RAG 是什么
- 为什么企业知识库需要 RAG
- 文档为什么要切分
- Embedding 和向量数据库分别做什么
- 如何减少 RAG 幻觉
- 企业 RAG 如何做权限隔离

## 9.1 RAG 是什么

RAG 可以简单理解为：

```text
先查资料，再让 AI 根据资料回答。
```

大模型本身不知道你的企业内部资料。

比如用户问：

```text
我们公司的报销流程是什么？
```

模型不知道你公司的制度，所以系统需要：

```text
先从企业文档里找相关内容
再把内容交给模型回答
```

## 9.2 为什么企业需要 RAG

企业资料通常包括：

- 员工手册
- 产品文档
- 客服话术
- 合同模板
- 培训资料
- 操作手册
- 规章制度

这些资料不在模型训练数据里，或者不是最新版本。

RAG 可以解决：

- 私有知识问答
- 最新资料问答
- 回答可追溯
- 减少模型胡说

## 9.3 RAG 完整流程

```text
上传文档
-> 解析文档
-> 清洗文本
-> 文本切分
-> 生成 Embedding
-> 存入向量数据库
-> 用户提问
-> 问题生成 Embedding
-> 检索相关片段
-> 拼接 Prompt
-> 大模型生成回答
-> 展示答案和引用来源
```

## 9.4 文档解析

不同文件需要不同解析方式：

```text
txt：直接读取
markdown：保留标题结构
PDF：提取页面文本
Word：提取段落
扫描件：先 OCR
网页：提取正文
```

解析后要保存：

- 文档名
- 文件类型
- 页码
- 标题
- 原文内容

## 9.5 文本切分

为什么要切分？

因为长文档不能整篇塞给模型。

原因：

- 上下文有限
- 成本高
- 检索不精准

常见切分方式：

### 固定长度切分

例如每 500 字切一段。

优点：简单。

缺点：可能切断语义。

### 按段落切分

适合普通文档。

### 按标题切分

适合 Markdown、制度文档、说明书。

### 带重叠切分

相邻片段保留一部分重叠，避免上下文断裂。

## 9.6 Embedding 是什么

Embedding 是把文本变成向量。

例如：

```text
员工年假怎么算？
```

会被转换成一串数字。

向量的作用是：

```text
让系统可以比较两段文字的语义相似度。
```

这样即使用户问：

```text
年休假规则是什么？
```

也能找到：

```text
员工年假计算方式...
```

## 9.7 向量数据库是什么

向量数据库用于保存和检索向量。

每条数据通常包含：

```text
向量
原文片段
文档 ID
知识库 ID
用户 ID
页码
标题
```

企业项目中，向量 payload 必须保存权限字段。

例如：

```json
{
  "user_id": "user_001",
  "knowledge_base_id": "kb_001",
  "document_id": "doc_001",
  "chunk_id": "chunk_001",
  "content": "员工入职满一年后可享受年假"
}
```

## 9.8 检索策略

最小版本：

```text
问题 Embedding -> 向量数据库 top_k 检索
```

进阶版本：

```text
向量检索
+ 关键词检索
+ Rerank
+ 相似度阈值
+ 权限过滤
```

## 9.9 RAG Prompt 模板

```text
你是企业知识库助手。
请只根据【资料】回答【问题】。
如果资料中没有答案，请回答“资料中没有找到相关信息”。
不要编造资料之外的信息。

【资料】
{context}

【问题】
{question}
```

## 9.10 引用来源

企业 RAG 必须展示引用来源。

引用来源包括：

- 文档名
- 页码
- 片段内容
- 相似度分数

作用：

- 用户可以验证答案
- 降低幻觉风险
- 方便追溯资料

## 9.11 如何减少 RAG 幻觉

方法：

- Prompt 要求只根据资料回答
- 资料不足时允许拒答
- 设置相似度阈值
- 展示引用来源
- 对关键场景人工审核
- 避免塞入太多无关资料

## 9.12 企业权限隔离

企业 RAG 最大风险之一是数据泄露。

必须保证：

```text
A 用户不能检索到 B 用户的文档
A 部门不能访问 B 部门知识库
```

做法：

- 文档表保存 `user_id` 或 `tenant_id`
- 向量 payload 保存 `user_id` 或 `tenant_id`
- 检索时加过滤条件
- 后端统一做权限校验

## 9.13 常见坑

### 坑 1：只做向量检索，不做权限过滤

这会导致严重数据泄露。

### 坑 2：chunk 太大

检索不精准，Token 成本高。

### 坑 3：chunk 太小

上下文不完整，回答容易断章取义。

### 坑 4：没有引用来源

用户无法判断答案是否可信。

### 坑 5：资料不足还强行回答

这会导致模型胡说。

## 本章总结

你只需要记住：

```text
RAG = 检索 + 大模型回答。
企业 RAG 的核心不是只把文档塞给模型，而是做好解析、切分、检索、引用、权限和拒答。
```
